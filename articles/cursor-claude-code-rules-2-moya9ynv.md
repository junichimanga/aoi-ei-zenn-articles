# Cursor/Claude Code の `.rules` ファイルをチーム運用する技術【無料サンプル2個公開】

## この記事でわかること

- `.cursor/rules`（および Claude Code の `.claude/rules`）ファイルがなぜチーム開発で重要なのか
- コピペして即使える無料サンプルルール2つ（命名規約ポリシー・新規参加者オンボーディングチェックリスト）
- ルールファイルを「形骸化させない」ための運用設計のポイント

---

## AIコーディングで「再現性のある出力」が出ない理由

Cursor や Claude Code は、プロンプトだけでもある程度使えてしまう。
だからこそ、チームで使い始めると**「人によって全然違うコードが出てくる問題」**に直面しやすい。

たとえばこういうケース。

- Aさんが生成したコードは `camelCase`、Bさんは `snake_case`
- 型定義ファイルを `src/types/` に置く人と `src/models/` に置く人が混在
- `// TODO` のフォーマットが統一されておらず、誰がいつ書いたか追跡不能

これらは「AIが悪い」のではなく、**AIへの指示が個人依存になっている**のが原因だ。

`.cursor/rules` や `.claude/rules` ファイルはこの問題への直接的な回答になる。プロジェクトルートに置くだけで、そのリポジトリ内でのAI生成コードにルールを「常時適用」できる仕組みだ。

問題は、「どう書けばいいか分からない」「テンプレートが手元にない」という点にある。

## サンプル1：命名規約・コメントポリシーのルール（Next.js 15 + TypeScript 向け）

以下は、商品パックに収録している `.cursor/rules` の冒頭セクションから抜粋したものだ。プレースホルダ（`{{ }}` 部分）を埋めるだけで使い始められる。

```
# ============================================================
# PROJECT RULES — Next.js 15 + Supabase + tRPC ECサイト
# プロジェクト名 : {{ PROJECT_NAME }}
# 担当者         : {{ OWNER }}
# 最終更新       : {{ YYYY-MM-DD }}
# コメント言語   : {{ COMMENT_LANG }}（ja / en / ja+en）
# ============================================================

## 言語・コメント方針
- コード内コメントは {{ COMMENT_LANG }} で書く
- TODOコメントは `// TODO(担当者名 YYYY-MM-DD): 内容` 形式を使う
- 自明なコードにコメントを付けない（WHY だけ書く）
- 公開 API のドキュメントは英語で書く

## 命名規約
- 変数・関数名は英語。ローマ字禁止
- ブール値は `is_` / `has_` / `can_` プレフィックスを付ける
- マジックナンバーは必ず定数化し、コメントで単位と根拠を明記する
  例: const MAX_CART_ITEMS = 99 // カート上限（UX仕様 #123 参照）
```

このようにルール自体に「なぜそのルールなのか」を埋め込んでおくと、AIが生成コードの根拠を説明しやすくなる。チームレビューの手間が減る可能性がある、というのが筆者の体感だ。

## サンプル2：新規メンバー向け3日間オンボーディングチェックリスト

ルールファイルを「書いて終わり」にしないための仕組みも重要だ。
パックには、チームへのルール定着を助けるオンボーディングテンプレートも含まれている。

```markdown
**1日目（セットアップ・読み込み）**
□ .cursor/rules を clone 後すぐに開いて全文を読んだ
□ ルールに疑問点があれば Draft PR でコメントした
□ サンプルタスクで AI 出力が生成されることを確認した

**2日目（実践・確認）**
□ PR を出し、ルール遵守チェックボックスを自分でチェックした
□ レビュアーのコメントとルールの対応関係を確認した

**3日目（フィードバック・定着）**
□ 1日目の Draft PR（ルール追記提案）をチームにレビュー依頼した
□ 月次棚卸し会の次回日程をカレンダーに追加した
□ 「このルールの意図が分からない」リストを1件以上 Issues に起票した
□ 担当メンターに3日間のフィードバックを伝えた
```

このチェックリストは PR テンプレートや Notion のオンボーディングドキュメントにそのまま貼り付けて使える。「ルールを読ませる」より「ルールに従って動かせる」設計にすることで、定着率が上がる傾向がある（筆者のチームでの体験として）。

## 使い方・カスタム方法

上記2つのサンプルは、`.cursor/rules` ファイルに直接コピーするか、`CLAUDE.md` に貼り付けるだけで有効になる。プレースホルダの `{{ PROJECT_NAME }}` などは、リポジトリごとに実際の値に置き換えること。命名規約はフレームワークや言語によって変わるため、たとえば Rails プロジェクトであれば `snake_case` ベースに書き換えるといい。チームで導入する場合は、最初から「完璧なルール」を目指すより、**まず5〜6行の最小ルールから始めて月次で育てる**運用が筆者には合っていた。
